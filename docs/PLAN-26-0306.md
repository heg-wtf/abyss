# Plan: QMD 통합 (마크다운 지식 검색)

## 문제 인식

cclaw의 현재 기억 시스템에는 근본적 한계가 있다:

1. **MEMORY.md 전체 주입**: 세션 시작 시 MEMORY.md 전체를 프롬프트에 넣음. 기억이 쌓일수록 토큰 낭비가 심해지고 컨텍스트 한계에 부딪힘
2. **대화 기록 검색 불가**: conversation-*.md 파일이 쌓이지만 과거 대화를 검색할 방법이 없음. "지난주에 뭐 물어봤지?"에 답할 수 없음
3. **외부 지식 접근 불가**: 사용자의 개인 노트, 문서, 미팅 메모를 봇이 참조할 방법이 없음

## QMD란?

[QMD](https://github.com/tobi/qmd)는 마크다운 문서를 로컬에서 인덱싱하고 검색하는 미니 검색 엔진이다 (Tobi Lütke 제작).

- **설치**: `npm install -g @tobilu/qmd`
- **검색 3모드**: BM25(키워드, ~30ms), Vector(의미, ~2s), Deep(확장+재정렬, ~10s)
- **Collection 기반**: 디렉토리를 collection으로 등록하여 인덱싱
- **로컬 처리**: SQLite + FTS5 + 벡터 임베딩, 모델 자동 다운로드 (~/.cache/qmd/models/)
- **MCP 서버**: stdio (`qmd mcp`) 또는 HTTP 데몬 (`qmd mcp --http --daemon`, 포트 8181)
- **데몬 관리 내장**: `qmd mcp --http --daemon` (시작), `qmd mcp stop` (중지)

### MCP 도구 (6개)

| 도구 | 설명 | 속도 |
|------|------|------|
| `search` | 키워드 기반 BM25 검색 | ~30ms |
| `vector_search` | 의미 기반 유사도 검색 | ~2s |
| `deep_search` | 쿼리 확장 + BM25 + 벡터 + 재정렬 | ~10s |
| `get` | 문서 전체 내용 조회 (경로/docid) | 즉시 |
| `multi_get` | 복수 문서 일괄 조회 (glob/목록) | 즉시 |
| `status` | 인덱스 상태, collection 목록, 문서 수 | 즉시 |

## 핵심 설계 결정: HTTP 데몬 방식

### stdio vs HTTP 비교

```
[stdio 방식 — 채택하지 않음]
메시지마다:
  Claude Code → "qmd mcp" 프로세스 생성 → ML 모델 로딩(수 초) → 검색 → 프로세스 종료
  → QMD를 안 쓰는 메시지에서도 MCP 서버 스폰 → 모든 메시지가 느려짐

[HTTP 데몬 방식 — 채택]
cclaw 시작 시:
  "qmd mcp --http --daemon" 한 번 실행 → 모델 로딩(한 번) → 포트 8181에서 대기

메시지마다:
  Claude Code → HTTP 요청 → QMD 데몬이 즉시 응답 (모델 이미 로딩됨)
  → QMD를 안 쓰는 메시지에는 오버헤드 없음 (HTTP 연결만, 프로세스 스폰 없음)
```

### Claude Code MCP HTTP 설정

Claude Code는 `.mcp.json`에서 HTTP/SSE 타입 MCP 서버를 지원한다:

```json
{
  "mcpServers": {
    "qmd": {
      "type": "http",
      "url": "http://localhost:8181/mcp"
    }
  }
}
```

→ subprocess를 스폰하지 않고, 이미 떠 있는 HTTP 엔드포인트에 연결

### QMD 데몬 lifecycle 관리

QMD가 자체 데몬 관리를 제공하므로, cclaw는 subprocess를 직접 관리할 필요 없다 (bridge와 달리 Popen/pipe 관리 불필요):

```python
# bot_manager.py에서
async def _start_qmd_daemon():
    """QMD가 이미 떠 있으면 스킵. 없으면 시작."""
    # 1. 봇 중 qmd 스킬을 사용하는 봇이 있는지 확인
    # 2. 없으면 스킵 (QMD 불필요)
    # 3. health check: http://localhost:8181 연결 시도
    # 4. 이미 떠 있으면 스킵
    # 5. subprocess.run(["qmd", "mcp", "--http", "--daemon"])
    # 6. health check 재시도 (최대 30초)

def _stop_qmd_daemon():
    """QMD 데몬 중지."""
    subprocess.run(["qmd", "mcp", "stop"])
```

```
cclaw start
  → start_bridge()          # Node.js bridge (기존)
  → _start_qmd_daemon()     # QMD HTTP 데몬 (신규, qmd 스킬 사용 시만)
  → start_polling()         # Telegram polling

cclaw stop
  → stop_polling()
  → _stop_qmd_daemon()      # QMD 데몬 중지
  → stop_bridge()           # Bridge 중지
```

### 1.5 대화 기록 자동 인덱싱

#### setup 시 자동 collection 등록

`cclaw skills setup qmd` 실행 시:

```python
# skill.py 또는 별도 qmd_setup.py

def setup_qmd_collections(bot_names: list[str]) -> None:
    """QMD에 cclaw 대화 기록 collection을 자동 등록한다."""
    import subprocess

    cclaw_home_path = cclaw_home()

    # 모든 봇의 sessions 디렉토리를 하나의 collection으로 등록
    sessions_path = cclaw_home_path / "bots"
    if sessions_path.exists():
        subprocess.run(
            ["qmd", "collection", "add", str(sessions_path),
             "--name", "cclaw-conversations",
             "--mask", "**/conversation-*.md"],
            capture_output=True,
        )

    # 인덱싱 실행
    subprocess.run(["qmd", "update"], capture_output=True)

    # 벡터 임베딩 생성 (시간이 걸림, 백그라운드 가능)
    subprocess.run(["qmd", "embed"], capture_output=True)
```

#### collection 네이밍

| Collection | 경로 | Glob | 자동/수동 |
|-----------|------|------|-----------|
| `cclaw-conversations` | `~/.cclaw/bots/` | `**/conversation-*.md` | 자동 (setup 시) |
| 사용자 노트 | 사용자 지정 | 사용자 지정 | 수동 (`qmd collection add`) |

#### 재인덱싱 전략

대화 파일은 매일 새로 생성되므로 주기적 재인덱싱이 필요하다:

**방안 A: cron 연동 (추천)**
```
cclaw cron add <bot> "매일 새벽 4시에 qmd update && qmd embed"
```
→ 기존 cron 시스템 활용, 추가 코드 불필요

**방안 B: heartbeat 연동**
- heartbeat 실행 시 `qmd update` 같이 실행
- heartbeat가 이미 주기적으로 돌고 있으므로 자연스러움
- 단, `qmd embed`는 시간이 걸려서 heartbeat에 넣기엔 무거울 수 있음

**방안 C: cclaw start 시 자동 update**
- `_start_qmd_daemon()` 에서 데몬 시작 후 `qmd update` 실행
- 가장 간단하지만, 시작이 느려질 수 있음

→ **Phase 1에서는 방안 A(cron) 권장**. 문서에 cron 설정 예시를 안내.

### 1.6 bot_manager.py 변경

```python
# _run_bots() 안에서

async def _run_bots(bot_names: list[str] | None = None) -> None:
    # ... 기존 로직 ...

    # Bridge 시작 (기존)
    bridge_started = await start_bridge()

    # QMD 데몬 시작 (신규)
    qmd_needed = _any_bot_uses_qmd(bots_to_run)
    if qmd_needed:
        await _start_qmd_daemon()

    # ... polling 시작 ...


def _any_bot_uses_qmd(bot_configs: list[dict]) -> bool:
    """봇 중 하나라도 qmd 스킬을 사용하는지 확인."""
    for config in bot_configs:
        if "qmd" in config.get("skills", []):
            return True
    return False


async def _start_qmd_daemon() -> bool:
    """QMD HTTP 데몬을 시작한다. 이미 떠 있으면 스킵."""
    import shutil

    if not shutil.which("qmd"):
        logger.warning("QMD not found, skipping daemon start")
        return False

    # Health check: 이미 떠 있는지 확인
    if await _qmd_health_check():
        logger.info("QMD daemon already running")
        return True

    # 데몬 시작
    result = subprocess.run(
        ["qmd", "mcp", "--http", "--daemon"],
        capture_output=True, text=True,
    )

    if result.returncode != 0:
        logger.error("QMD daemon start failed: %s", result.stderr)
        return False

    # 시작 대기 (최대 30초)
    for _ in range(30):
        if await _qmd_health_check():
            logger.info("QMD daemon started on port 8181")
            return True
        await asyncio.sleep(1)

    logger.error("QMD daemon did not become ready")
    return False


async def _qmd_health_check() -> bool:
    """QMD HTTP 엔드포인트 연결 확인."""
    try:
        reader, writer = await asyncio.wait_for(
            asyncio.open_connection("localhost", 8181),
            timeout=2,
        )
        writer.close()
        await writer.wait_closed()
        return True
    except (ConnectionRefusedError, asyncio.TimeoutError, OSError):
        return False


def _stop_qmd_daemon() -> None:
    """QMD 데몬 중지."""
    import shutil

    if shutil.which("qmd"):
        subprocess.run(["qmd", "mcp", "stop"], capture_output=True)
        logger.info("QMD daemon stopped")
```

## Phase 2: RAG 기반 메모리 (MEMORY.md 검색 주입)

### 현재 흐름 (Phase 1까지)

```
세션 시작 시:
  GLOBAL_MEMORY.md 전체 주입
  + MEMORY.md 전체 주입
  + 최근 20턴 대화
  + 새 메시지
```

→ MEMORY.md가 커지면 매 세션 부트스트랩에 불필요한 토큰 소비

### Phase 2 흐름

```
세션 시작 시:
  GLOBAL_MEMORY.md 전체 주입 (작으므로 유지)
  + QMD로 MEMORY.md에서 현재 메시지와 관련된 기억만 검색 (상위 5개)
  + 최근 20턴 대화
  + 새 메시지
```

### 구현 개요

1. **MEMORY.md를 QMD collection으로 등록**
   - collection: `cclaw-memory`
   - 경로: `~/.cclaw/bots/`
   - glob: `**/MEMORY.md`

2. **세션 부트스트랩 변경** (`handlers.py._prepare_session_context`)
   - QMD 스킬이 활성화되어 있으면:
     - 새 메시지를 쿼리로 QMD search 실행
     - MEMORY.md 전체 대신, 검색 결과 상위 N개만 주입
   - QMD 스킬이 없으면: 기존 방식 유지 (호환성)

3. **QMD 검색 호출 방법**
   - Python에서 직접 QMD HTTP API 호출 (requests/httpx)
   - 또는 `qmd search "query" --json` CLI 호출
   - Claude Code를 거치지 않음 (부트스트랩 단계이므로)

### 변경 파일

- `handlers.py`: `_prepare_session_context()`에 QMD 검색 분기 추가
- `session.py`: `load_bot_memory()` 대신 `search_bot_memory(query)` 추가
- 새 모듈 `qmd_client.py`: QMD HTTP API 직접 호출 (MCP 아닌 REST)

### 위험 요소

- QMD 데몬이 죽어있으면 fallback 필요 (MEMORY.md 전체 주입으로 복귀)
- 검색 품질이 낮으면 중요한 기억을 놓칠 수 있음
- "항상 기억해야 할 것" vs "관련될 때만 꺼낼 것" 구분 필요

### 복잡도: 높음

Phase 1이 안정화된 후 진행. MEMORY.md가 실제로 커져서 토큰 문제가 발생할 때.

## 기술적 고려사항

### QMD HTTP 엔드포인트 검증 필요

mcp.json의 URL 경로를 확인해야 한다:
- `http://localhost:8181/mcp` (HTTP transport)
- `http://localhost:8181/sse` (SSE transport)
- QMD 소스 코드에서 실제 경로 확인 필요

**검증 방법**: `qmd mcp --http` 실행 후 엔드포인트 테스트

### 포트 충돌

QMD 기본 포트 8181이 다른 서비스와 충돌할 수 있다.

해결: `skill.yaml`에 포트 설정 추가하거나, 환경 변수 `CCLAW_QMD_PORT` 지원.

```yaml
# skill.yaml 확장
qmd_port: 8181  # 또는 environment_variable_values에 추가
```

### QMD 의존성 무게

| 항목 | 크기/시간 |
|------|-----------|
| qmd CLI (npm) | ~수 MB |
| ML 모델 (자동 다운로드) | ~100MB+ |
| 첫 embed 실행 | 문서 수에 따라 수 분 |
| SQLite 인덱스 | 문서 수에 비례 |

→ 다른 skill(naver-cli 등)보다 무거움. 설치 가이드에 명시.

### 프라이버시

- 모든 인덱싱/검색은 로컬 (클라우드 전송 없음)
- 대화 기록에 민감 정보 포함 가능 → 로컬 전용이지만 사용자에게 안내
- QMD 인덱스 파일: `~/.cache/qmd/index.sqlite` (삭제 시 재인덱싱 필요)

### Fallback

QMD 데몬이 죽어있거나 연결 불가 시:
- Phase 1: Claude Code가 MCP 연결 실패를 처리 (QMD 도구 사용 불가, 에러 메시지)
- Phase 2: MEMORY.md 전체 주입으로 fallback (기존 방식)
